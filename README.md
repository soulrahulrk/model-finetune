# Vedaz Astrology Assistant — Qwen3-8B Fine-Tuning Project

Production-grade fine-tuning pipeline for a Vedic-astrology chat assistant, built on the
real "Chat Data for Assessment of Applicants" dataset (55 raw conversations, Hindi/Hinglish/
English, safety-tuned "Vedaz AI Vedic astrologer" persona). This repo covers the full lifecycle:
dataset audit → cleaning/augmentation → QLoRA fine-tuning of **Qwen3-8B** with Unsloth →
export (safetensors + GGUF) → vLLM production serving on a VPS → evaluation.

### Data pipeline (reproduces everything under `data/processed/`)

```bash
cd src/data_prep

# 1. Parse the raw (non-standard, concatenated-JSON) file, validate, dedupe, produce stats
python analyze_and_clean.py \
  --input ../../data/raw/chat_data_for_assessment_of_applicants.json \
  --out-jsonl ../../data/processed/clean_dataset.jsonl \
  --out-report ../../data/processed/analysis_report.json

# 2. Build the 5 manual + 100 synthetic conversation files (deterministic, no external calls)
python ../../data/manual_conversations/_build_manual_conversations.py
python ../../data/synthetic/_build_synthetic_conversations.py

# 3. Merge all 3 sources into a stratified 90/10 train/val split
python split_dataset.py \
  --inputs ../../data/processed/clean_dataset.jsonl \
           ../../data/synthetic/synthetic_astrology_100.jsonl \
           ../../data/manual_conversations/manual_conversations.jsonl \
  --out-dir ../../data/processed \
  --val-ratio 0.10 --seed 42
```

Expected result: 50 real + 100 synthetic + 5 manual = **155 unique conversations** → **139
train / 16 val**, with every source represented in both splits (see
[`docs/01_dataset_analysis.md` §9](docs/01_dataset_analysis.md#9-preprocessing-strategy-executive-summary)
for why per-source stratification matters here).

## Training guide

```bash
cd src/training
python train_unsloth.py --config training_config.yaml
```

Requires a CUDA GPU — see [`docs/02_finetuning_strategy.md` §2](docs/02_finetuning_strategy.md#2-hardware-requirements)
for VRAM sizing (QLoRA on Qwen3-8B fits in ~7-9GB; a single RTX 4090/L4/A10G 24GB is the
recommended target). Override any hyperparameter without editing the YAML:

```bash
python train_unsloth.py --config training_config.yaml --num_train_epochs 4 --learning_rate 1e-4
```

This produces a LoRA adapter at `outputs/qwen3-8b-vedaz-adapter/`. Merge and export:

```bash
python merge_and_export.py --config training_config.yaml
# --skip-gguf to skip the (slower) GGUF quantization step during iteration
```

Outputs:
- `outputs/qwen3-8b-vedaz-merged-fp16/` — full safetensors model, what vLLM serves (Task 2)
- `outputs/qwen3-8b-vedaz-adapter/` — LoRA adapter only (small, portable)
- `outputs/qwen3-8b-vedaz-gguf/` — `q4_k_m` and `q8_0` GGUF quantizations for llama.cpp/Ollama

Every design decision (why QLoRA over full-FT/LoRA, why rank 16, why 3 epochs, why packing is
off, why loss is masked to assistant tokens only, etc.) is justified against this specific
dataset's measured properties in
[`docs/02_finetuning_strategy.md`](docs/02_finetuning_strategy.md) — read that before changing
`training_config.yaml`, since most values there are not arbitrary defaults.

## Inference guide

```bash
cd src/inference

# Single prompt
python infer.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 \
  --prompt "Mera career kab tak stabilize hoga?"

# Interactive chat
python infer.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 --interactive

# Batch (used by the evaluation scripts — generates against every held-out val example)
python infer.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 \
  --batch-in ../../data/processed/val.jsonl --batch-out eval_outputs/predictions.jsonl
```

Falls back from Unsloth's fast-inference path to plain `transformers.generate` automatically if
Unsloth isn't installed in the current environment — useful for sanity-checking the merged
export on a machine that only has the serving stack, not the training stack.

## Evaluation guide

```bash
cd src/evaluation

python eval_text_metrics.py --predictions eval_outputs/predictions.jsonl --output eval_outputs/text_metrics.json
python eval_perplexity.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 --val-file ../../data/processed/val.jsonl --output eval_outputs/perplexity.json
python eval_safety.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 --probes safety_probes.jsonl --output eval_outputs/safety_report.json
python eval_hallucination.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 --probes hallucination_probes.jsonl --output eval_outputs/hallucination_report.json
python eval_bias.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 --output eval_outputs/bias_report.json
python eval_prompt_robustness.py --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 --output eval_outputs/robustness_report.json
python benchmark.py --backend transformers --model ../training/outputs/qwen3-8b-vedaz-merged-fp16 --output eval_outputs/bench_results.json
```

Why there are 7 separate scripts and no single "quality score": explained in
[`docs/04_evaluation_guide.md`](docs/04_evaluation_guide.md) — this is a safety-sensitive,
open-ended generative task where averaging metrics together hides exactly the failures (a
category-specific safety-guardrail miss, a demographic bias disparity) that matter most for a
ship/no-ship decision.

## Troubleshooting guide

| Symptom | Cause | Fix |
|---|---|---|
| `json.decoder.JSONDecodeError: Extra data` reading the raw dataset | The raw file is 55 concatenated JSON objects, not a JSON array or JSONL | Use `analyze_and_clean.py`, not `json.load()` directly — see [`docs/01_dataset_analysis.md` §1](docs/01_dataset_analysis.md#1-dataset-format) |
| `CUDA out of memory` during training | `per_device_train_batch_size` too high, or another process holds GPU memory | Check `nvidia-smi`; the default config (batch 2, grad-accum 4) fits ~7-9GB — see [`docs/02_finetuning_strategy.md` §2](docs/02_finetuning_strategy.md#2-hardware-requirements) |
| `ModuleNotFoundError: unsloth` when just reading `--help` | Expected — training/inference scripts import Unsloth lazily so `--help` works without a GPU environment | Install the training stack (`requirements.txt`) before actually running training/merge/inference |
| Training loss not decreasing after epoch 1 | Learning rate or rank mismatch for this dataset size | Don't casually increase epochs/rank on a 155-example dataset — re-read [`docs/02_finetuning_strategy.md` §6](docs/02_finetuning_strategy.md#6-why-not-full-fine-tuning-restated-as-a-decision-record) on overfitting risk first |
| GGUF export hangs or is very slow | Expected — llama.cpp conversion + quantization is CPU-bound and slow for larger quant sets | Use `--skip-gguf` during iteration; only export GGUF for a final checkpoint |
| vLLM server 502s from Nginx right after deploy | Model still loading (30-90s for an 8B model) | Wait for the `Uvicorn running` log line before routing traffic — see [`docs/03` §18](docs/03_vllm_vps_hosting_guide.md#18-common-errors-and-debugging) for the full error table |
| Streaming responses hang through Nginx | Missing `proxy_buffering off` | Confirm `deployment/nginx/qwen.conf` is actually the file loaded (`nginx -t`, check `sites-enabled`) |
| Safety eval shows category-specific failures | Expected use of the tool — that's what `eval_safety.py`'s per-category breakdown is for | Do not ship on an "overall pass rate" alone; see [`docs/04` §7](docs/04_evaluation_guide.md#7-safety-evaluation) |
| BERTScore/eval scripts download a large model on first run | `bert-base-multilingual-cased` is fetched from HF Hub on first use | Expected one-time cost (~700MB); ensure `huggingface-cli login` is set up if the environment has restricted/no anonymous HF access |

## Future improvements

- **Grow the real-data pool.** 50 real conversations is a strong signal for what "good" looks
  like but is fundamentally small; if more real user consultations become available, re-run the
  full pipeline — the 90/10-per-source split and QLoRA rank/epoch choices in
  `docs/02_finetuning_strategy.md` were explicitly tuned for this data scale and should be
  revisited as it grows (e.g., rank could increase, full fine-tuning could become viable at
  10K+ examples).
- **Expand the safety probe suite.** `safety_probes.jsonl` (30 probes, 9 categories) and
  `hallucination_probes.jsonl` should grow continuously as new failure modes are discovered in
  production — treat both as living test suites, not one-time deliverables.
- **Add an LLM-judge pass to CI.** `eval_safety.py --use-llm-judge` is implemented but requires
  an API key not available in this environment; wiring it into a CI gate (block merges on a
  category pass-rate regression) is the natural next step before any production rollout.
- **Multi-turn training data at scale.** Only 1 real conversation and the 5 manual consultations
  currently model long multi-turn dialogue (§2 of the dataset audit found 78% of real data is
  single-turn); if the product surfaces long consultations often, prioritize collecting/generating
  more multi-turn examples over more single-turn breadth.
- **A/B the base model choice empirically.** The Qwen3-8B vs. Qwen2.5-7B decision in
  `docs/02_finetuning_strategy.md` §1 is reasoned from published benchmarks and the dataset's
  language-mix profile; once both are actually fine-tuned on this data, running the full
  evaluation suite (§ above) against both would let the choice be validated empirically rather
  than by reasoning alone.
- **Quantized serving.** `docs/03` §17-18 mentions AWQ/GPTQ as a memory/cost optimization but the
  pipeline currently only exports fp16 safetensors + GGUF; adding an AWQ export step to
  `merge_and_export.py` would let `vllm serve --quantization awq` be used directly for
  higher-throughput production serving.
- **Continuous evaluation on production traffic** (with consent/anonymization) — the current
  suite is all offline/synthetic probes; sampling real (anonymized) production conversations for
  periodic human evaluation would catch drift the fixed probe sets can't.
